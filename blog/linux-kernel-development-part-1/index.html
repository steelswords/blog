Building the Raspberry Pi Kernel from Source

‚ÄúFrom source?‚Äù you say. ‚ÄúBut Tristan, we can totally just build a module as long as we have the headers.‚Äù Yes, yes. That‚Äôs true. But we‚Äôre on a learnin‚Äô adventure here today, so we‚Äôre going to start by building the kernel from source. Plus, building the kernel from source and then building kernel modules off of that is a great way to ensure the module is compiled against the same version as the kernel is running, which is Very Important.

First, I clone the sources and run a build, to make sure I have everything I need.

We need to make the config file that controls the kernel build process. Because we‚Äôre doing this on my (much more capable) machine instead of on the target, we include some extra parameters to specify how to cross-compile.

I put my laptop into ‚ÄúHigh performance‚Äù mode for whatever that does and run the following:

Now it‚Äôs time to deploy our changes. Plug in that SD card and let‚Äôs get down to it. This step is a little involved, so I wrote up a script to do it for me.

deploy.sh:

#!/usr/bin/env bash

set -ue -o pipefail

DESTINATION_ROOT="/media/tristan/rootfs"
DESTINATION_BOOT="/media/tristan/boot"

KERNEL="${KERNEL-kernel}"

if ! [[ -d "$DESTINATION_ROOT" ]]
then
    echo "Could not find root destination \"$DESTINATION_ROOT\". Is it mounted?"
    exit 1
fi

if ! [[ -d "$DESTINATION_BOOT" ]]
then
    echo "Could not find boot destination \"$DESTINATION_BOOT\". Is it mounted?"
    exit 2
fi

# Install kernel modules to SD card
sudo env PATH=$PATH make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- INSTALL_MOD_PATH="$DESTINATION_ROOT" modules_install

# Backup old img
cp "${DESTINATION_BOOT}/$KERNEL.img" "${DESTINATION_BOOT}/$KERNEL-backup.img"

# Install new kernel and device tree files
cp arch/arm/boot/zImage "${DESTINATION_BOOT}/$KERNEL.img"
cp arch/arm/boot/dts/*.dtb "${DESTINATION_BOOT}/"
cp arch/arm/boot/dts/overlays/*.dtb* "${DESTINATION_BOOT}/overlays/"
cp arch/arm/boot/dts/overlays/README "${DESTINATION_BOOT}/overlays/"

Now we‚Äôll slide that SD card in the Pi Zero W and boot ‚Äòer up.

Success! It still boots. That‚Äôs good news. Now let‚Äôs make some oh-so-satisfying changes.

Let‚Äôs Make a Kernel Module

These first few ‚Äúlearning‚Äù modules won‚Äôt do anything serious, so I don‚Äôt feel inclined to put them in their own git repo(s). Instead, let‚Äôs make a new git branch and a new directory in the kernel repo.

git checkout -b my_custom_development
mkdir custom_modules
cd custom_modules

Then, let‚Äôs add our beginners‚Äô kernel module and a Makefile. As you can see, I am a fan of silly messages for test printouts. It makes it more fun.

Now for helloworld.c:

#include <linux/init.h>
#include <linux/module.h>
MODULE_LICENSE("GPLv3");

static int hello_world_init(void)
{
    printk(KERN_ALERT "Hello, world! This is My First Kernel Module(TM) by Playskool(TM)\n");
    return 0;
}

static void hello_world_exit(void)
{
    printk(KERN_ALERT "Goodbye, world! Remember to tip your waitresses\n");
}

module_init(hello_world_init);
module_exit(hello_world_exit);

And the Makefile:

CR_C := arm-linux-gnueabihf-
KERNEL_DIR := /home/tristan/Repos/raspberrypi-linux/
obj-m += helloworld.o

all:
	make ARCH=arm CROSS_COMPILE=$(CR_C) -C $(KERNEL_DIR) M=$(shell pwd) modules

clean:
	make ARCH=arm CROSS_COMPILE=$(CR_C) -C $(KERNEL_DIR) M=$(shell pwd) clean

The Makefile has two targets: all and clean. Let me break down the make commands for you.

  • ARCH=arm sets the architecture to arm. Important for cross-compiling.
  • CROSS_COMPILER=$(CR_C): Sets the cross-compiler binary prefix. When make calls gcc now, it will call arm-linux-gnueabihf-gcc instead.
  • -C $(KERNEL_DIR): This has make change directory to KERNEL_DIR before it does anything else. We do this so all the headers and other kernel-ly goodness is in place before we make.
  • M=$(shell pwd) This sets the variable M, which is typically used in kernel Makefiles to represent the directory of an external module to build.
  • modules: The actual target to build. All the logic for how this is executed is defined in the main kernel Makefile.

So now we have a bunch of files in our custom_modules directory. Let‚Äôs copy one over to our Raspberry Pi Zero W and load it up to see the fruits of our labor! We need to copy it to somewhere in /lib/modules/

scp helloworld.ko pizero:/tmp
ssh pizero

Now we‚Äôll run some commands on the Pi Zero. First we copy the kernel module to an appropriate directory. Then, we‚Äôll run depmod and modprobe, to load our module into the kernel. Then, we‚Äôll check to make sure it‚Äôs loaded with lsmod. Finally, we look at the message we wanted our module to print using journalctl.

On the Pi:

sudo cp /tmp/helloworld.ko /lib/modules/5.15.84+/kernel/
modinfo /lib/modules/5.15.84+/kernel/helloworld.ko
depmod
sudo modprobe helloworld
lsmod | grep helloworld
journalctl -xe

There it is! That‚Äôs our message. When we unload the module, we will see the other message as well. That was a lot of fun! Tune in next time for a less trivial kernel module as we work our way towards building drivers.
